Teknisk oversikt

Kort dokumentasjon for hvordan Lilletorget drift er bygget opp, hvor data kommer fra, og hvor feil normalt sjekkes først.

Applikasjon

  • NavnFibaro10 / Lilletorget drift
  • Lokal prodhttp://192.168.20.218:8110 på QNAP.
  • Ekstern prodhttps://fibaro10.onrender.com. Brukes når løsningen skal nås utenfra.
  • KodeGitHub-repo lewinordby/fibaro10. Produksjon bygges fra main.
  • BackendFastAPI med PostgreSQL-database, server-renderte sider og GZip-komprimering.
  • FrontendServer-renderte Jinja-maler. Felles design ligger i static/app.css, meny/header i static/owner-nav.css.
  • BrandingLilletorget-logo. Sol/P-symbol åpner/lukker meny, tekstlogo går til Status.

Tilgang

  • InnloggingBrukernavn og passord/nøkkel. Normal bruk skjer med cookie etter innlogging.
  • RollerMaster, Innstillinger og Vanlig bruker.
  • MasterKan opprette brukere, se nøkler, endre roller og abonnere på tilgangsvarsel.
  • InnstillingerKan endre styringsverdier for lys og ventilasjon, men ikke administrere brukere.
  • LoggerLokal/QNAP-baserte tjenester kan poste med egen bruker via HTTP-headere.

Dataflyt

HC3

Poster lys, lux, ventilasjon, temperatur, energi og Yr-grunnlag til API-et.

Lokale loggere

QNAP/Docker henter Roborock, SUN2 og EasyPark-data og sender strukturerte batcher inn.

Manuelle importer

Elvia-timesfil lastes opp fra Energi. Område/navn kan suppleres via nettleserutvidelser.

Database

Data lagres i egne tabeller for lys, ventilasjon, renhold, soling, energi, parkering, AI, tilgang og importstatus.

Grensesnitt

Sidene viser status, tidslinjer, logger, grafer, innstillinger og feilsignaler fra samme datagrunnlag.

Hyppige logger

  • LysLux og lysstatus hvert 5. minutt. Hendelser ved statusendring.
  • VentilasjonTemperaturer, viftestatus, modus og effektgrunnlag hvert 5. minutt.
  • Energi HC3Realtime W og akkumulert kWh hvert minutt fra quickapps i HC3.
  • YrHentes/lagres når varselet må fornyes, og knyttes til ventilasjonsgrunnlaget.

SUN2-import

  • DagsfilerNattlig nedlasting/import av summer per rom og dato.
  • EnkeltimerSkrapes omtrent hvert 5. minutt for inneværende dag/måned. Duplikater unngås med naturlig nøkkel.
  • MedlemmerSynkes periodisk. SUN2-id er den stabile nøkkelen.
  • SengerRom-/sengmetadata gir stabil fysisk rom-id selv om SUN2-navn endres.

Renhold og energi

  • RoborockLokal logger sender robotstatus, jobber, planer, forbruksdeler og kartdata.
  • ElviaManuell opplasting av timesfil under Energi → Elvia.
  • Forbruk/sengBeregnes fra SUN2 enkelttimer og HC3 differanseforbruk når nøyaktig én seng er i bruk.
  • HC3 verktøyLokalt vedlikeholdsverktøy: HC3 vedlikehold.

Parkering

  • KildeEasyPark-data hentes av lokal downloader og importeres til parkeringsmodulen.
  • OppdateringDaglig jobb henter automatisk. Knappen i Parkering → Oversikt kan trigge manuell oppfrisking.
  • DedupliseringImporten oppdaterer eksisterende parkeringer når samme post kommer på nytt, slik at pågående parkeringer kan få sluttid senere.
  • KjøretøyRegnr normaliseres og kobles mot kjøretøytabell med tellinger, beløp, første/siste sett og SVV-status.
  • BerikelseMerke, modell, årsmodell, farge og klasse lagres i kjoretoy_nokkeldata når data finnes.
  • Egne feltNavn, Sun2id, notat og område ligger i kjoretoy. Område brukes for sted/statistikk, navn for intern kobling.

Oversikten fokuserer på valgt dag: pågående parkeringer først, deretter avsluttede. Kjøretøyvisninger viser nå navn, sted under biltype der det finnes.

Nettleserutvidelser

  • Områdebrowser_extensions/parking_name_lookup henter liste over biler uten område og kan skrive område tilbake via API.
  • Navnbrowser_extensions/parking_manual_name brukes når navn legges inn manuelt og lagres på samme kjøretøy-API.
  • BatchOmrådeutvidelsen kan hente opptil 1000 regnr fra Fibaro10 når brukeren har nødvendig rettighet og lovlig tilgang hos kilden.
  • SikkerhetUtvidelsene lagrer lokal konfigurasjon i nettleseren. Fibaro10 tar imot via innlogget bruker eller API-headers.

Drift og deploy

  • QNAP deployQNAP-instansen oppdateres fra GitHub main og bygges med docker compose -f docker-compose.qnap.yml up -d --build fibaro10.
  • Render deployRender kjører samme repo/branch og kan trigges automatisk på commit eller manuelt via Render API.
  • Helsesjekk/health viser at appen svarer og hvilke datatabeller som finnes.
  • LoggerContainerlogger viser postinger fra HC3, Roborock, SUN2, EasyPark/importstatus og eventuelle 500-feil.
  • Uten authDe fleste sider krever innlogging. /health, /favicon.ico og innloggingssiden er åpne.

Oversiktssider

  • StatusSamler nåstatus, siste hendelser og datakildestatus.
  • LysDagslogg er standard underside og viser lux, lysperioder og sammenligning mot dagen før.
  • VentilasjonDagslogg er standard underside og kan vise flere temperaturer og viftestreker.
  • SolingDagslinje viser fysisk romaktivitet. Statistikksider bruker aggregerte tabeller for ytelse.
  • EnergiStatus viser timeforbruk fra HC3 og Elvia når import finnes. Forbruk/seng beregnes fra sikre enkeltperioder.
  • ParkeringOperativ dagsoversikt, historikk, statistikk, områdeoppslag og kjøretøydetaljer.

Feilsøking først

  • 1Åpne Status → Datakilder og se om noe er tregt, gammelt eller feilet.
  • 2Sjekk Status → Dashboard for Driftsradar og siste hendelser.
  • 3Sjekk aktuell fagmeny: Lys, Ventilasjon, Renhold, Soling eller Energi.
  • 4Ved importfeil: sjekk lokal logger/QNAP først, deretter API-status i appen. For parkering: se Parkering → Oversikt og EasyPark-status.
  • 5Ved rare tider: kontroller om kilden leverer lokal tid eller UTC. SUN2 og Elvia vises som kildetid, HC3/Yr konverteres til Europe/Oslo ved visning.
  • 6Ved feil design etter deploy: sjekk at både QNAP og Render serverer ny static/app.css, siden disse kan være ulike miljøer.
